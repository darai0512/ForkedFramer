# 整合性チェックと欠損メディア対応

## 背景（問題意識）

- 画像/動画の保存が長時間になると、タブ切替・ページ離脱・凍結/破棄などで書き込みが中断される可能性がある。
- その結果、`.bin` が不完全なまま残り、後からの読み込みでデコード失敗や `AbortError` / `EncodingError` を誘発しうる。
- `mediaResource` が `null` になり得る経路があるため、UIが `null` を想定していない箇所でクラッシュしやすい。

関連: 事象の背景と一次対応は [documents/sentry-aborterror-investigation.md](documents/sentry-aborterror-investigation.md) を参照。

## 戦略

- 破損/欠損を**早期に検出**して、後続処理に持ち込まない
- UIとデータの両面で**影響を局所化**し、アプリ全体を落とさない
- 破損検知時の**復旧/再取得の導線**を用意する
- **再現性のある検証手段**を確保し、手戻りを減らす
- `mediaResource` は **null を許容しない**。代わりに **「欠損」状態**を明示的に導入し、専用UIで表現する
- 欠損時は **フィルムをスキップせず**、「欠損メディアのフィルム」として保持する

## 戦術

### 1. 整合性チェック

- **ハッシュ（SHA-256等）**: サイズ一致でも破損を検出したい場合。書き込み/読み込みのコスト増。

### 2. 復旧/隔離の方針

- 破損検出 → 該当blob削除 → UIは欠損扱い（placeholder）
- 破損検出 → DBの `blobPath` を無効化 → 再保存/再取得を促す

### 3. 欠損状態の導入（nullを使わない）

- `mediaResource` は **null 非許容**に統一する
- 代替として **「欠損（missing）」状態**を新設し、型で明示する
- UIは **欠損専用の表示**（placeholder/再取得導線）を持つ
- 欠損時は **フィルムを破棄しない**（レイアウト保持のため）

### 4. Vitestでカバーできる範囲

- **imagePacking の欠損保持**: 破損/欠損で `loadMediaFunc` が missing を返す場合、該当フィルムが欠損状態で保持されること
- **null排除の型整合**: `mediaResource` が null を返さない型設計が破綻していないこと（コンパイル/型チェック）
- **読込失敗時の扱い**: `readMediaResource()` が失敗した場合のフォールバック経路（missing 化・エラー通知の分岐）
- **dump/undump**: 欠損を含むデータがあっても処理が継続できること（例外で停止しない）

### 5. 手動検証（詳細マニュアルは別文書）

手動テスト手順は別文書に分離:  
[documents/manual-missing-media-tests.md](documents/manual-missing-media-tests.md)

### 6. 今後の予定

1. **欠損状態の仕様化**: `mediaResource` に missing を導入し、null を廃止する
2. **UIの欠損表示**: missing 専用の表示と再取得導線を設計
3. **挙動統一**: 破損検知時の処理（削除/無効化/通知）の基準を決定
4. **検証導線**: 欠損データ作成の簡易手段を整備し、再現性を確保
